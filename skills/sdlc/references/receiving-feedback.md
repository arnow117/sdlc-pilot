# receiving-feedback — 接收意见并修复的纪律(蒸馏自 superpowers:receiving-code-review)

> 共享纪律,被任何**有 fix-cycle 的地方**引用:`sdlc-build`(任务自检 / 调试修复)、`sdlc-review`(fix-first 处置)。
> 治的病:**agent 一被指出问题就疯狂乱改 / 表演式同意 / 盲目照单全收**。
> 核心原则:**先核实再改、先问再假设、技术正确 > 社交舒适。意见是待评估的建议,不是必须执行的命令。**

## 响应模式(每条意见都走一遍)

```
READ        完整读完,先别反应
UNDERSTAND  用自己的话复述这条要求(说不清就问)
VERIFY      对照代码库现实核实——这条对本仓成立吗?
EVALUATE    技术上对吗?会破坏现有功能吗?当前实现是不是有它的理由?
RESPOND     技术性确认,或有理由地反驳
IMPLEMENT   一次一项,每项各自验证
```

## 三条防乱改铁律

1. **改前先核实,不盲改**:任何 finding 先对照代码库确认它**真的成立 + 对本仓正确**,再动手。看着对 ≠ 真的对。
2. **不清楚就 STOP,先把所有点澄清完再动**:部分理解 = 错误实现。"我懂 1/2/3/6,4/5 需澄清"——不要边问边改前面的。
3. **一次一项、各自验证、验无回归**:绝不一把梭改一堆。顺序:阻塞性(break/安全)→ 简单(typo/import)→ 复杂(重构/逻辑)。

## 该反驳就反驳(意见也会错)

遇到以下情况**用技术理由反驳,不照单全收**(也不防御性顶撞):
- 会破坏现有功能 / reviewer 缺全貌 / 技术上对本栈不对 / 有兼容性·legacy 理由 / 与已定架构决策冲突。
- **YAGNI grep**:意见让"做个正经的 X"→ 先 grep 它是否真被调用;**没人用就提议删,别顺手加**没要求的东西。
- 自己反驳错了 → 一句话事实纠正即可:"核实了你对,我之前理解错在 X,改。"——不长篇道歉、不辩解。

## 禁止 / 改用

| ❌ 禁止 | ✅ 改用 |
|---|---|
| "你说得对!""好的我改"(表演式同意) | 复述技术要求,或直接动手 |
| 改前不核实就实现 | 先 VERIFY 对照代码库 |
| 批量改不逐项测 | 一次一项,各自验 |
| 改对了写"谢谢指出""好眼力" | 直接说"已修:<改了什么>",代码本身证明你听进去了 |

## 在 sdlc 里的接入点
- **sdlc-build**:任务收尾两阶段自检(spec→quality)发现问题、调试子循环定位到 bug → 按本纪律修(核实→一次一项→验),不一被指出就乱改。
- **sdlc-review**:Fix-First 处置每条 finding 前,先按本纪律核实该 finding 真成立、对本仓正确,再 AUTO-FIX;拿不准的 ASK,不盲修。
